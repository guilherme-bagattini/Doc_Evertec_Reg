# Decisões fechadas para o MVP do EvertecReg

Este arquivo registra o que ficou mais claramente definido para a primeira etapa do produto.

## Fechado para o MVP

- O fluxo inicial sera conservador: gravar primeiro, aprovar depois.
- Nao foi prevista a possibilidade de voltar uma aprovacao ja feita.
- Nao foi prevista segregacao complexa de permissoes nesta etapa.
- O status de erro na interface sera generico; o diagnostico detalhado fica nos logs.
- Nao foi previsto versionamento de publicacoes no MVP.
- Nao foi previsto dashboard operacional no MVP.
- A IA nao vai aprender de forma adaptativa no MVP.
- A tabela e editada por conteudo, e nao como bloco massivo.
- Justificativa no descarte faz sentido e deve ser considerada.

## Fica para incremento ou validacao posterior

- Dashboard com curadorias pendentes, erros e notificacoes.
- Alertas para novos itens a aprovar ou publicados.
- Versionamento das publicacoes.
- Segregacao de acesso por perfis mais detalhada.
- Mecanismos de aprendizagem da IA com base em historico.
- Relevancia por tema, como BACEN, em visoes mais ricas para o cliente.

## Como interpretar isso

O MVP parece focado em garantir um fluxo seguro de operacao, rastreabilidade e leitura tecnica, sem tentar resolver de imediato todos os recursos de observabilidade e automacao avancada.

Isso reduz risco de escopo e mantém a entrega mais centrada no que a equipe ja conseguiu validar com o time de negocio.
